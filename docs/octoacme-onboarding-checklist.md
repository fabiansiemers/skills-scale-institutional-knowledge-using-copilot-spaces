## Purpose
Ensure new team members have the context, access, and resources needed to contribute effectively to OctoAcme projects.

## When to use
Whenever a new team member joins a project or team, regardless of their role.

## Onboarding Checklist

### First Day
- [ ] Welcome meeting with Project Manager
- [ ] Introduction to team members and their roles
- [ ] Access to project repository and communication channels
- [ ] Review [Project Management Overview](octoacme-project-management-overview.md)
- [ ] Review [Roles & Personas](octoacme-roles-and-personas.md) to understand responsibilities
- [ ] Receive project-specific context (charter, current sprint goals)

### First Week
- [ ] Review relevant process documentation for your role
  - Developers: [Execution & Tracking](octoacme-execution-and-tracking.md)
  - Product/Project Managers: [Project Initiation](octoacme-project-initiation.md) and [Planning](octoacme-project-planning.md)
  - UX Designers: Design system documentation and accessibility standards
  - Technical Architects: Architecture Decision Records (ADRs) and technical standards
  - All roles: [Risk Management & Communication](octoacme-risks-and-communication.md)
- [ ] Attend sprint ceremonies (standup, planning, review, retrospective)
- [ ] Shadow a team member in your role (if available)
- [ ] Set up local development environment (for technical roles)
- [ ] Review recent retrospective notes to understand team improvements
- [ ] Understand the project's success metrics and current status

### First Month
- [ ] Contribute to at least one deliverable (PR, design, analysis, etc.)
- [ ] Participate in code/design review process
- [ ] Attend cross-functional meetings relevant to your role
- [ ] Provide onboarding feedback to improve the process
- [ ] Understand escalation paths and blocker resolution process
- [ ] Review project risk register and understand key risks
- [ ] Complete any role-specific training or certifications

## Role-Specific Onboarding

### Developers
- [ ] Clone repository and verify build/test environment
- [ ] Review branching strategy and PR conventions
- [ ] Understand CI/CD pipeline and quality gates
- [ ] Review code style guide and linting rules
- [ ] Complete first PR (small bug fix or documentation update)

### Product Managers
- [ ] Review product roadmap and backlog
- [ ] Meet with key stakeholders
- [ ] Understand success metrics and measurement approach
- [ ] Review customer feedback channels
- [ ] Attend customer/user research sessions

### Project Managers
- [ ] Review project timeline, milestones, and dependencies
- [ ] Understand resource allocation and budget
- [ ] Review risk register and mitigation plans
- [ ] Set up status reporting cadence
- [ ] Meet with cross-team dependencies

### UX Designers
- [ ] Review design system and component library
- [ ] Understand accessibility standards and testing process
- [ ] Access design tools and collaboration platforms
- [ ] Review user research findings and personas
- [ ] Understand design review and feedback process

### Scrum Masters / Agile Coaches
- [ ] Review team's agile maturity and current practices
- [ ] Understand team's retrospective action items
- [ ] Review team health metrics and velocity trends
- [ ] Meet with team members individually
- [ ] Understand stakeholder expectations

### Technical Architects
- [ ] Review architecture documentation and ADRs
- [ ] Understand technical debt and architectural roadmap
- [ ] Review security and performance requirements
- [ ] Identify technical risks and dependencies
- [ ] Schedule architecture review sessions

### Customer/End-User Advocates
- [ ] Access user feedback channels and support tickets
- [ ] Review user personas and journey maps
- [ ] Understand user pain points and feature requests
- [ ] Schedule user interviews or observation sessions
- [ ] Understand how feedback is prioritized

### Data Analysts
- [ ] Access analytics platforms and data sources
- [ ] Review current KPIs and measurement frameworks
- [ ] Understand experiment design process
- [ ] Review past A/B test results and learnings
- [ ] Set up reporting dashboards

## Onboarding Success Criteria
- Team member can independently complete core tasks in their role
- Team member knows who to ask for help in different situations
- Team member understands project goals and their contribution
- Team member feels welcomed and psychologically safe

## Continuous Improvement
- Collect onboarding feedback after 30 days
- Update this checklist based on new team member experiences
- Share learnings in retrospectives
